# 水墨屏纪念日相册 · 软件方案

小程序 + Node 后端（含图像处理模块）：业务全流程、绑定逻辑、MQTT 规范、图像处理、数据流转。

> 本文档为「软件 / 服务端 / 小程序」部分，硬件与固件显示层见 [硬件方案.md](./硬件方案.md)，总览见 [README.md](./README.md)。

## 一、整体业务全流程

### 阶段 1：上电联网，获取唯一 MAC
1. ESP32 上电启动，屏幕显示自身 WiFi 配网名称（AP 配网），手机连接后为其配置网络。
2. 联网成功后，读取本机固定 MAC 地址。
3. 通过 MQTT 上报 MAC 到 Node，请求绑定。

### 阶段 2：Node 判断绑定状态
- **未绑定** → 进入阶段 3（生成并显示二维码）。
- **已绑定** → 跳过二维码，直接进入阶段 6（推送最近一条纪念日画面）。

> ✅ 补充点 1：二维码仅在「未绑定」状态出现，用于首次绑定；绑定完成后永不重复生成二维码。

### 阶段 3：生成绑定二维码（仅首次）
1. Node 根据 MAC 生成二维码内容（二维码内容 = 纯 MAC，尽量短以保证墨水屏可扫）。
2. Node 图像模块把「二维码图像」量化（纯黑白二值即可）并打包成点阵字节数组。
3. 得到点阵字节后进入下发流程（生成二维码内容、渲染、量化、打包全部在 Node 内完成）。

> ✅ 补充点 2：二维码内容拼装、图像渲染、量化打包都由 Node 完成，无外部图像服务。

### 阶段 4：下发点阵，水墨屏显示二维码
1. Node 通过 MQTT 下发点阵到设备下行主题 `device/{MAC}/render`。
2. 因整屏点阵可能超过 MQTT 单包上限，采用**分包下发 + 帧序号 + 完整性校验**。
3. ESP32 收齐分包后拼帧，刷新水墨屏显示二维码。

### 阶段 5：小程序扫码绑定
1. 用户打开小程序扫码，读取 MAC。
2. 小程序把「openid + MAC」提交 Node 接口。
3. Node 存库完成绑定（`openid ↔ MAC` 一对一）。
4. 绑定成功后小程序展示该设备，可管理纪念日。

> 说明：一个用户对应一台设备。若需换绑，在设备上**按钮三超长按 8s 恢复出厂**，重新配网后屏幕会再次显示绑定二维码，用小程序重新扫码绑定即可。

### 阶段 6：推送纪念日文字画面（日常态）
1. 用户在小程序设置 / 修改纪念日（名称、日期、倒计时），提交 Node。
2. Node 根据绑定关系找到设备 MAC，**在服务端把纪念日文字渲染成 800×480 位图**（Canvas 画字）。
3. Node 图像模块将该位图做六色量化 + 抖动，打包成点阵字节数组。
4. Node 通过 MQTT 下发到 `device/{MAC}/render`（同样分包）。
5. ESP32 收帧刷新水墨屏，显示纪念日文字画面。

> 说明：中文字体渲染在**服务端**用 Canvas 完成，ESP32 不做字体渲染；量化打包也在 Node 图像模块内完成。

## 二、设备识别 / 绑定核心逻辑（无冲突）

1. **唯一凭证**：ESP32 硬件 MAC（不可篡改、换 WiFi 不变）。
2. **通信隔离**：MQTT 主题以 MAC 区分，多台设备互不串消息。
3. **绑定关系**：数据库存 `openid ↔ MAC` **一对一**（一个用户对应一台相框）。
4. **换绑方式**：设备上**按钮三超长按 8s（恢复出厂）** → 重新配网 → 屏幕重新显示二维码 → 小程序重新扫码绑定。

## 三、MQTT 主题规范（统一）

| 方向 | 主题 | 用途 |
| --- | --- | --- |
| 上行 | `device/{MAC}/up` | 上报 MAC、状态、在线心跳、`sync` 拉取请求 |
| 下行 | `device/{MAC}/render` | 下发点阵（二维码 / 纪念日画面），分包 |
| 下行 | `device/{MAC}/ack` | 可选；`sync_ok` / `sync_has_update` 等轻量应答（也可合入 `up` 双向） |

**上行 `sync` 请求示例（按钮二长按触发）**

```json
{ "type": "sync", "contentVersion": 12 }
```

**Node 处理逻辑**

1. 比对设备上报的 `contentVersion` 与库中该 MAC 的最新版本。
2. 若一致 → 回复 `{"type":"sync_ok"}`，设备绿灯闪 1 次，不刷屏。
3. 若服务端更新 → 走 `render` 分包下发，帧头携带新 `contentVersion`；设备刷成功后更新本地版本号。

## 四、Node 图像处理模块说明（不使用 Python）

图像处理不再独立成 Python 服务，而是 Node 服务内的一个模块，纯 JS 实现。核心算法可**直接复用现有示例**（见第六节），风险低。

**推荐依赖（npm）**

| 用途 | 库 | 说明 |
| --- | --- | --- |
| 生成二维码图像 | `qrcode` | 内容 = 纯 MAC，输出到 Canvas / PNG |
| 文字渲染成位图 | `@napi-rs/canvas` | 预编译二进制，`getImageData()` 接口与浏览器一致，示例 JS 几乎零改动；需 `registerFont` 加载中文字体 |
| 图片解码 / 缩放到 800×480 | `sharp` | 可选；`@napi-rs/canvas` 也能直接绘制 |
| MQTT 收发 | `mqtt` | 分包下发点阵 |

**处理流水线（都在 Node 内）**
1. **画布**：用 `@napi-rs/canvas` 生成 800×480 画布，绘制二维码或纪念日文字。
2. **取像素**：`ctx.getImageData()` 拿到 RGBA 像素 buffer。
3. **量化 + 抖动**：把像素映射到 E6 六色调色板（二维码用黑白两色即可）。
4. **打包**：六色模式 **每像素 4bit、2 像素 / 字节**，整屏 `800×480/2 = 192000` 字节。
5. **分包**：切成小包，经 MQTT 下发到 `device/{MAC}/render`。

> 职责边界：Node 一手包办业务逻辑、MQTT、小程序接口、图像渲染/量化/打包；ESP32 只接收字节并刷屏。

**优势**：单一语言、单一运行时，无 `child_process`、无 Python 环境依赖，部署最轻。

## 五、数据流转链路汇总

**首次绑定（含二维码，仅一次）**
```
ESP32 联网 → 上报 MAC → Node 判定未绑定
        → Node 生成二维码内容(纯 MAC) → Node 图像模块 量化/打包点阵
        → MQTT 分包下发 → 水墨屏显示二维码
        → 小程序扫码上传(openid+MAC) → Node 存库绑定
```

**日常推送（只推纪念日，无二维码）**
```
小程序编辑纪念日 → Node 渲染文字位图 → Node 图像模块 量化/打包点阵
        → MQTT 分包下发 device/{MAC}/render → ESP32 拼帧刷新水墨屏
```

**设备主动拉取（按钮二长按，小程序模式）**
```
ESP32 上行 sync(contentVersion) → Node 比对版本
        → 有更新：MQTT 分包 render → ESP32 拼帧刷新
        → 无更新：ack sync_ok → 绿灯闪 1 次
```

**蓝牙临时传图（按钮一切蓝牙模式 + 按钮二长按开广播）**
```
手机 BLE 连接传图 → ESP32 收字节写缓存并刷屏（蓝灯快闪）
        → 不改变 Node 侧 contentVersion；切回小程序模式后 sync 恢复正式画面
```

## 六、参考示例代码指引（图像处理算法）

现成的官方蓝牙传图示例可直接作为实现参考，路径：`docs/7.3寸E6蓝牙传图-示例代码/`。

### 6.1 图像处理算法（可直接移植到 Node 图像模块）
文件：`docs/7.3寸E6蓝牙传图-示例代码/蓝牙传图增强版.html`（纯浏览器 JS，逻辑可原样搬到 Node）

- **取像素**：`ctx.getImageData()`（约 `:722`）→ Node 用 `@napi-rs/canvas` 的同名接口替代。
- **抖动**：`ditherImage()`（约 `:900`），支持 `sixColor / fourColor / threeColor / blackWhiteColor` 四种调色板，可调饱和度/亮度/对比度/扩散。
- **量化 + 打包**：`processImageData()`（约 `:1041`）。六色模式核心：

```javascript
const byte = (colorIndex1 << 4) | colorIndex2; // 每像素 4bit，2 像素 1 字节
```

- **颜色索引**（须与固件一致）：黑=0 白=1 黄=2 红=3 蓝=5 绿=6（跳过索引 4）。完整对照见 [硬件方案.md](./硬件方案.md) 的颜色索引对照表。
- **整屏字节数**：`800 × 480 / 2 = 192000` 字节。

> 迁移要点：把 HTML 里 `ditherImage` / `processImageData` 两个纯函数拷到 Node，输入改成从 `@napi-rs/canvas` 或 `sharp` 得到的 RGBA buffer 即可，其余逻辑不用改。
