## 一、硬件基础

- **主控**：ESP32-S3 **N16R8**（16MB Flash + 8MB PSRAM，自带 WiFi），Arduino 框架开发。
  - 8MB PSRAM 足够缓存整屏帧缓冲（800×480 多色），有利于拼帧后一次性刷屏。
- **屏幕**：7.3 寸 **E6 彩色电子墨水屏**，分辨率 **800×480**，六色（黑 / 白 / 红 / 黄 / 蓝 / 绿，Spectra 6）。
- **设备唯一标识**：出厂固定 MAC 地址（`WiFi.macAddress()`），换 WiFi 不改变。
- **状态指示**：板载 **WS2812 RGB LED**（`PIN_RGB_LED = 48`），用颜色与闪烁表达模式、进度与异常。
- **物理按键**：共 **3 颗**，分工为「模式 / 执行 / 系统」，无需第四颗键（当前方案为单张画面 + 蓝牙临时传图，非多图轮播相册）。

### 1.1 引脚分配总表

板卡参考：`docs/图片资料/esp32s3-N16R8.JPG`；墨水屏 SPI 接线参考：`docs/图片资料/水墨屏引脚.png`。

| 功能 | 宏定义 | GPIO | 排针位置（开发板丝印） | 说明 |
| --- | --- | --- | --- | --- |
| 墨水屏 MOSI | `PIN_EPD_DIN` | **9** | 左侧 | SPI 数据 |
| 墨水屏 SCLK | `PIN_EPD_SCLK` | **10** | 左侧 | SPI 时钟 |
| 墨水屏 CS | `PIN_EPD_CS` | **11** | 左侧 | 片选 |
| 墨水屏 DC | `PIN_EPD_DC` | **12** | 左侧 | 数据/命令 |
| 墨水屏 RST | `PIN_EPD_RST` | **13** | 左侧 | 复位 |
| 墨水屏 BUSY | `PIN_EPD_BUSY` | **14** | 左侧 | 忙信号 |
| 板载 RGB LED | `PIN_RGB_LED` | **48** | 右侧 | WS2812，状态指示 |
| **按钮一（模式键）** | `PIN_BTN_MODE` | **4** | 左侧 | 切换蓝牙 / 小程序模式 |
| **按钮二（执行键）** | `PIN_BTN_ACTION` | **5** | 左侧 | 刷新、拉取、BLE 广播 |
| **按钮三（系统键）** | `PIN_BTN_SYSTEM` | **6** | 左侧 | 长按配网 / 超长按出厂 |

> 以上引脚与 `platformio.ini` 中 `build_flags` 保持一致，固件通过 `-D` 宏注入，勿在代码里写死 GPIO 号。

### 1.2 按键选型说明

三颗按键均接在**左侧排针 GPIO 4 / 5 / 6**，理由如下：

1. **不与墨水屏冲突**：水墨屏已占用 GPIO 9–14，RGB LED 占用 GPIO 48。
2. **避开 Strapping 脚**：不使用 GPIO 0、3、45、46（影响上电启动模式）。
3. **便于外壳布线**：4 / 5 / 6 在左侧排针连续三根，旁边即有 GND，按键一端接对应 GPIO、另一端接 GND 即可。
4. **支持深度睡眠唤醒**：GPIO 4 / 5 / 6 均可作为 `ext1` 唤醒源，与第二节「任意键唤醒」一致。

**接线方式**：固件配置 `INPUT_PULLUP`，按下时 GPIO 被拉低至 GND（低电平有效）。

```
GPIO 4 ──┬── 按钮一 ── GND
GPIO 5 ──┼── 按钮二 ── GND
GPIO 6 ──┴── 按钮三 ── GND
```

板载 **BOOT** / **RST** 按键保留原厂功能（烧录与复位），不纳入本方案三键逻辑。

### 1.3 预留 / 勿用引脚

| GPIO | 原因 |
| --- | --- |
| 9–14 | 墨水屏 SPI |
| 48 | 板载 WS2812 RGB LED |
| 0、3、45、46 | ESP32-S3 Strapping，上电电平敏感 |
| 43、44 | USB Serial/JTAG（通常未引出排针） |

若后续扩展传感器或扩展口，可优先考虑右侧排针 **GPIO 15–18、21、35–42、47**（均未占用）。

## 二、按键与 LED 交互设计

三颗按键覆盖模式切换、刷新/拉取、配网重置；所有未列出的组合默认无效（防抖 50ms，短按 < 800ms，长按 ≥ 800ms，重置长按 3s，出厂超长按 8s）。

### 2.1 按键分工

| 按键 | 名称 | GPIO | 职责 |
| --- | --- | --- | --- |
| 按钮一 | 模式键 | `PIN_BTN_MODE` = **4** | 在「蓝牙传图模式」与「小程序推送模式」之间切换；当前模式写入 NVS，重启后恢复 |
| 按钮二 | 执行键 | `PIN_BTN_ACTION` = **5** | 按当前模式执行刷新、拉取、广播等；刷新进行中再次按下可**取消**当前任务 |
| 按钮三 | 系统键 | `PIN_BTN_SYSTEM` = **6** | 长按进入配网；超长按恢复出厂（清 WiFi + 绑定 + 模式偏好） |

### 2.2 模式键（按钮一）真值表

模式键**仅短按有效**，长按无效。每次切换后立即更新 LED 常亮色，并将 `workMode` 写入 NVS。

| 当前状态 | 短按（< 800ms） | 长按（≥ 800ms） | 刷新进行中短按 |
| --- | --- | --- | --- |
| 蓝牙传图模式（蓝灯常亮） | 切至**小程序推送模式**；绿灯常亮；写 NVS | 无效 | 允许切换；LED 立即变绿灯常亮；当前收包/刷屏**不中断**，直至完成或被按钮二取消 |
| 小程序推送模式（绿灯常亮） | 切至**蓝牙传图模式**；蓝灯常亮；写 NVS | 无效 | 允许切换；LED 立即变蓝灯常亮；当前收包/刷屏**不中断**，直至完成或被按钮二取消 |
| AP 配网中 | 无效 | 无效 | 无效 |
| 未绑定（显示二维码） | 无效 | 无效 | 无效 |

> **上电默认模式**：小程序推送模式（绿灯常亮）。从 NVS 读取到已保存的 `workMode` 时，以 NVS 值为准。

### 2.3 模式与待机 LED

| 模式 | 待机 LED | 说明 |
| --- | --- | --- |
| 蓝牙传图模式 | **蓝灯常亮** | 手机通过 BLE 手动传图并刷屏（参考软件方案示例） |
| 小程序推送模式 | **绿灯常亮** | 默认日常态；画面由 Node 经 MQTT 下发 |

### 2.4 执行键（按钮二）真值表

| 当前模式 | 短按（< 800ms） | 长按（≥ 800ms） | 刷新进行中再按 |
| --- | --- | --- | --- |
| 蓝牙传图（蓝灯） | 用**本地缓存**重刷上一张蓝牙图；无缓存则蓝灯快闪后红灯闪 2 次 | **进入 / 退出 BLE 广播**（等待手机连接传图）；广播中蓝灯慢闪 | **取消**当前收图/刷屏，回到蓝灯常亮 |
| 小程序推送（绿灯） | 用**本地缓存**重刷当前画面（消除残影或补救刷屏失败） | 经 MQTT 上行发 `sync` 请求，查服务端是否有新画面；有则下发，无则绿灯闪 1 次后恢复常亮 | **取消**当前收包/刷屏，回到绿灯常亮 |

> **小程序模式长按拉取**：固件向 `device/{MAC}/up` 发送 `{"type":"sync","contentVersion":<本地版本号>}`；Node 比对版本，若服务端更新则走 `device/{MAC}/render` 分包下发，否则回复 `sync_ok` 且无需刷屏。

### 2.5 系统键（按钮三）重置粒度

| 操作 | 时长 | LED | 效果 |
| --- | --- | --- | --- |
| 仅重配 WiFi | 长按 **3s** | 橙灯慢闪 → 进入 AP 配网 | 清除 WiFi 凭证，**保留**绑定关系；配网成功后按已绑定流程拉取最近画面 |
| 恢复出厂 | 超长按 **8s** | 橙灯快闪 → 红灯闪 3 次确认 | 清除 WiFi、绑定状态、模式偏好；重启后等同首次未绑定，屏幕重新显示绑定二维码 |

> 换绑设备：使用**恢复出厂**（超长按 8s），而非仅重配 WiFi。

### 2.6 非按键触发的 LED 状态

| 状态 | LED |
| --- | --- |
| 上电 / 连接 WiFi / 连接 MQTT | 白灯或黄灯慢闪 |
| 收包 / 刷屏进行中 | 当前模式色（蓝或绿）**快闪**，直至完成 |
| 成功完成 | 回到当前模式色常亮 |
| 失败（断网、MQTT 超时、收包校验失败、BLE 断开） | **红灯闪 3 次**，再回到模式色常亮 |
| 深度睡眠唤醒 | 任意键唤醒；先黄灯闪 1 次，再恢复模式色 |

深度睡眠下**任意按键**均可唤醒，无需单独唤醒键；唤醒后执行键短按可立即重刷缓存画面。

### 2.7 未绑定 / 配网中的按键策略

| 设备状态 | 按钮一 | 按钮二 | 按钮三 |
| --- | --- | --- | --- |
| AP 配网中 | 无效 | 无效 | 长按 3s 可重新开始配网 |
| 未绑定（显示二维码） | 无效 | 短按：重新请求二维码（MQTT 上报 + 等待 Node 下发） | 长按 3s：重配 WiFi；超长按 8s：恢复出厂 |

### 2.8 双通道内容优先级

- **小程序推送模式**：以 Node 下发的 `contentVersion` 为准；本地缓存最后一次成功刷屏的帧。
- **蓝牙传图模式**：蓝牙收到的图写入本地缓存并**立即显示**，但不改变 Node 侧正式 `contentVersion`（蓝牙图为临时预览，切回小程序模式后，执行键长按 `sync` 会恢复服务端正式画面）。
- 实现时须持久化：`workMode`、`contentVersion`（存 NVS）与 `lastFrame` 整帧数据（存 Flash 分区 / LittleFS，**不放 NVS**），详见第五节「NVS 与帧缓存存储」。

## 三、固件侧（ESP32 显示端参考）

目录：`docs/7.3寸E6蓝牙传图-示例代码/GDEP073E01/`

- 驱动：厂商裸驱动 `Display_EPD_W21.*`（GoodDisplay GDEP073E01 / E6 面板专属初始化序列，仅依赖 `SPI.h`），非 GxEPD2。
- 显示流程：接收字节 → `EPD_W21_WriteCMD(0x10)` 进入 DTM → 逐字节 `EPD_W21_WriteDATA` 写显存 → `EPD_autoSequence()` 上电/刷新/下电/睡眠（见 `GDEP073E01.ino`）。
- **注意**：示例把「接收通道」做成 **BLE**（512 字节/包）；本方案改为 **WiFi + MQTT** 下发。固件里"写显存 + 刷屏"部分可复用，"BLE 接收"部分需替换为 MQTT 订阅 + 分包重组。
- **已知小坑**：示例 `GDEP073E01.ino` 里打印 MAC 的 `uint8_t mac[6]` 声明后未赋值（没调用 `WiFi.macAddress()` / `esp_read_mac`），会打印随机值；本方案依赖 MAC，实现时务必真正读取一次。

## 四、刷新频率限制与防抖

E6 墨水屏**全刷耗时约 12–20s，且刷新次数有寿命上限**，必须在固件侧做频率限制，避免用户/服务端频繁触发导致刷屏排队、闪烁或损耗过快。

### 4.1 刷新最小间隔

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| `EPD_MIN_REFRESH_INTERVAL` | **60s** | 两次**实际全刷**之间的最小间隔；未到间隔的请求进入「待刷」而非立即执行 |
| `EPD_BUSY_TIMEOUT` | **30s** | 单次刷屏 `BUSY` 拉高的最长等待；超时判失败，红灯闪 3 次并下电 |

- **刷新中互斥**：任一时刻只允许一帧在刷。刷屏进行中收到的新帧只保留**最后一帧**（按 `contentVersion` 取最新，旧的丢弃），当前帧刷完后再刷最新的那一帧，中间的全部跳过。
- **相同帧跳过**：待刷帧的校验和 / `contentVersion` 与屏上当前帧一致时，直接跳过刷新（仅回 ACK，不点屏）。

### 4.2 内容防抖（合并抖动）

- 收到新帧后**不立即刷**，先启动一个 **`EPD_DEBOUNCE = 3s`** 的合并窗口；窗口内若又来新帧，重置计时并只保留最新帧。窗口结束后才真正刷一次，避免用户连续改动触发多次刷屏。
- 防抖窗口与最小间隔叠加：即"窗口结束"且"距上次刷屏 ≥ 最小间隔"两个条件同时满足才刷；否则继续等到间隔到达。

### 4.3 按键防抖

沿用第二节参数：GPIO 电平防抖 **50ms**，短按 `< 800ms`、长按 `≥ 800ms`。执行键短按重刷同样受 4.1 的最小间隔约束（间隔未到时绿/蓝灯闪 1 次提示"稍后"，不强刷）。

> 以上阈值均通过 `platformio.ini` 的 `-D` 宏注入，便于调参，勿写死。

## 五、NVS 与帧缓存存储

存储分两层：**小而关键的状态用 NVS**，**大块整帧数据用 Flash 文件系统**（NVS 不适合存 192KB 级别数据）。

### 5.1 NVS 键值（`Preferences`，命名空间 `jnr`）

| 键 | 类型 | 说明 |
| --- | --- | --- |
| `workMode` | `uint8` | 0 = 小程序推送模式，1 = 蓝牙传图模式；开机恢复 |
| `contentVersion` | `uint32` | 当前屏上正式画面的版本号，用于 `sync` 比对 |
| `frameCrc` | `uint32` | `lastFrame` 的 CRC32，用于开机/重刷时校验帧文件完整性 |
| `bound` | `uint8` | 是否已绑定（0/1），决定开机走二维码还是日常态 |
| `wifiSsid` / `wifiPass` | `string` | WiFi 凭证（也可交由底层 `WiFi` NVS 管理，二选一，避免两处不一致） |
| `lastRefreshTs` | `uint32` | 上次成功刷屏时间戳，用于第四节最小间隔判断 |

- **写入时机**：`workMode` 在模式键切换后立即写；`contentVersion` / `frameCrc` 在刷屏成功后写；避免高频写入以保护 NVS 擦写寿命。

### 5.2 整帧缓存（LittleFS / 独立 Flash 分区）

- `lastFrame` 为整屏点阵 **192000 字节（800×480÷2）**，存到 **LittleFS 文件**（如 `/last.bin`）或自定义 data 分区，**不进 NVS**。
- 断电保图靠墨水屏本身；`lastFrame` 的作用是**开机/唤醒后执行键短按可本地重刷**（消残影、补救失败），无需再联网拉取。
- 开机流程：读 `frameCrc` → 校验 `/last.bin` → 一致则可直接本地重刷；不一致则丢弃并按模式走联网拉取。
- 16MB Flash 分区建议：为 LittleFS 预留 ≥ 512KB（`platformio.ini` 选用带较大 SPIFFS/LittleFS 的分区表，如 `default_16MB.csv` 或自定义）。

## 六、自建配网页（设备自启 AP + 网页配网）

不依赖微信小程序配网，也不用厂商 SmartConfig；设备**自身开热点 + 内置网页**，用户手机连该热点后在网页里填 WiFi 完成配网（Captive Portal 强制门户）。

### 6.1 触发时机

- **首次上电且 NVS 无 WiFi 凭证**：自动进入配网。
- **系统键长按 3s**：主动重配 WiFi（见 2.5，保留绑定关系）。
- **连接已存 WiFi 连续失败**（如 3 次、约 30s）：回退进入配网。

### 6.2 配网流程

1. 设备切到 **AP 模式**开热点，SSID 形如 `JNR-Frame-{MAC后4位}`（无密码或固定弱密码），**屏幕显示该 AP 名称与"用手机连接后按提示配网"文案**。
2. 固件启动 **DNS 劫持（把所有域名解析到设备 IP，如 `192.168.4.1`）+ 内置 HTTP 服务**，实现 Captive Portal：手机连上后系统自动弹出配网页。
3. 配网页提供：**扫描到的 WiFi 列表 + 手动输入 SSID/密码**，提交到设备接口（如 `POST /save`）。
4. 设备用收到的凭证尝试连接：
   - **成功** → 写入 NVS（`wifiSsid`/`wifiPass`）→ 关闭 AP 与 HTTP → 转 STA 模式联网 → 走已绑定/未绑定后续流程。
   - **失败** → 网页返回错误，停留在配网页，橙灯继续慢闪，允许重填。
5. 配网期间 LED：**橙灯慢闪**；配网成功切模式色常亮。

### 6.3 实现要点

- 可用 `WiFiManager` 库，或 `WebServer` + `DNSServer` 自行实现 Captive Portal（后者更可控、便于把配网页做成中文并显示品牌）。
- 配网页为**单页 HTML**，直接嵌在固件（`PROGMEM` 字符串），无需外部资源，断网也能打开。
- **仅 STA 联网需求，无需 AP+STA 共存**：配网时用纯 AP，配网成功后切纯 STA，减少射频与内存压力。
- 配网页表单**只在本地局域网**提交，密码不出设备（不上云）。

## 七、BLE 与 WiFi 单射频切换（同一时刻只开一个）

ESP32-S3 的 WiFi 与 BLE **共用同一 2.4GHz 射频**，同时启用会互相抢占带宽、增大内存占用并降低稳定性。本方案**不做共存**，而是**跟随 `workMode` 二选一独占射频**，由模式键（按钮一）切换。

### 7.1 切换规则

| 目标模式 | 射频状态 | 关闭动作 | 开启动作 |
| --- | --- | --- | --- |
| 小程序推送模式（绿灯） | **仅 WiFi/MQTT**，BLE 关闭 | `esp_bluedroid_disable()` → `esp_bt_controller_disable()`（释放 BT 内存可 `esp_bt_controller_mem_release(ESP_BT_MODE_BLE)`） | 启动 WiFi STA + MQTT 订阅 |
| 蓝牙传图模式（蓝灯） | **仅 BLE**，WiFi 关闭 | `mqtt_disconnect()` → `WiFi.disconnect(true)` → `WiFi.mode(WIFI_OFF)` | 初始化 BLE、按执行键长按开广播等连接 |

### 7.2 切换时序（模式键短按）

1. 立即更新 LED 为目标模式色常亮，并把 `workMode` 写入 NVS。
2. **优雅收尾旧射频**：若正在收包/刷屏，遵循 2.2「当前任务不中断」——先让当前帧刷完（或被执行键取消），再关闭旧射频协议栈。
3. 关闭旧射频 → 短暂延时（等控制器释放）→ 初始化并启用新射频。
4. 新射频就绪后：小程序模式自动重连 MQTT；蓝牙模式待执行键长按再开广播。
5. 切换过程中如失败（连不上 WiFi / BLE 初始化失败）：红灯闪 3 次，回到目标模式色常亮并允许重试。

### 7.3 注意事项

- **开机按 `workMode` 只初始化对应一种射频**，不要两个都起。
- BLE 与 WiFi 控制器的 enable/disable 要成对、按顺序调用，避免内存泄漏或 controller 状态错乱。
- 深度睡眠前应先关闭当前射频再进睡眠；唤醒后按 `workMode` 重新初始化对应射频。
- 因两者不共存，**蓝牙传图时收不到 MQTT 推送**属预期行为；切回小程序模式后由执行键长按 `sync` 补齐最新画面（见 2.4 / 2.8）。
